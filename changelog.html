Development History

EVERY VERSION.
EVERY DECISION.

Tidemark wasn't designed in a vacuum. Each version was a direct response to a real failure — in privacy, in reliability, in trust. This is the complete record of how the engine was built.

v1.0 Foundation

The concept proved.

First working report. The original question was simple: can a spoken, messy account from a staff member become a structured, defensible incident record without a lawyer in the room?

check_small 15 incident playbooks covering core hospitality event types
check_small Single-pass AI extraction — one prompt, full field set
check_small Basic regex PII detection — names, rooms, phone numbers
check_small Structured JSON output, SHA-256 report anchoring
v2.0 Privacy Rebuilt

Regex wasn't enough.

Pattern matching catches predictable names. Real incidents don't have predictable names. A guest named something the regex had never seen would pass through unredacted. That was unacceptable. The entire privacy layer was rebuilt.

check_small Structural NLP (Compromise.js) replaces regex — catches names by linguistic context, not pattern
check_small Identity maps moved entirely to volatile RAM — no disk writes at any stage of the process
check_small Token restoration pipeline — real names swapped back into the final report after AI processing
check_small 16th playbook added — General catch-all for uncategorised incident types
v2.1 Human in the Loop

The AI needed context. Not names.

A critical problem emerged: the AI could see that [PERSON_1] was involved in the incident, but it couldn't know whether that person was a guest, a staff member, or a witness. Role context changes everything about how a report reads. The solution was to pause and ask — without revealing who the person is.

check_small CLARIFICATION phase introduced — system detects person tokens and pauses before extraction
check_small Operator classifies each detected person as Guest, Staff, or Witness — AI receives the role, never the name
check_small Clarification queue — multiple names processed in sequence, not in a single disorienting list
v2.2 Extraction Engine

One pass wasn't enough.

Asking a 4-billion parameter model to extract 20 fields in a single call is asking it to fail. It loses track. It produces incomplete JSON. It hallucinates values it can't find. The extraction engine was redesigned from a single call into a multi-stage pipeline where each pass has one focused job.

check_small Discovery pass — pre-scan maps the narrative terrain before any field extraction begins
check_small 8 semantic field groups processed independently — datetime, location, personnel, severity, emergency response, actions, cause, documentation
check_small Retry passes for unresolved fields — reruns each group with increased inferential allowance before asking the operator
check_small Per-field confidence scoring — every extracted value carries a 0.0–1.0 confidence score; anything below 0.50 is discarded
v2.3 Neural Layer

NLP alone has a blind spot.

Structural NLP identifies names by grammar rules. It works well in clean, predictable text. Real incident reports are not clean. Names appear without titles, without context, mid-sentence — in patterns no grammar rule was written for. A neural model was added to catch what structural rules miss, and a consensus gate was introduced to prevent false positives from either layer acting alone.

check_small BERT-Small ONNX model added — runs locally on device CPU, zero cloud inference for PII detection
check_small Statistical anomaly filter added — 10,000-word dictionary prevents common words from being falsely redacted
check_small Consensus gate — structural NLP alone is sufficient to redact; neural model requires statistical corroboration. Three layers. One decision.
check_small Zero-trust linguistic rules — 300+ common hospitality terms explicitly whitelisted to prevent operational language from being redacted
v2.4 Current Release

The version that ships.

The engine was stable. The privacy layer was hardened. The remaining gap was reach — the night auditor alone in the lobby doesn't have a laptop. They have a phone. This version brings Tidemark to where incidents actually happen.

check_small iOS native app — SwiftUI, built for in-the-moment capture from the floor, not a desktop
check_small PDF and DOCX export — insurance adjusters and legal teams work in documents, not JSON
check_small Improved name detection for edge cases — staff listed without titles, names embedded mid-sentence
check_small Confidence audit trail — per-field confidence metadata stored with every completed report
v3.0 In Development

The next problem.

Android support. Multi-property management. Report review workflows for GMs who need to approve before export. The engine works. Now it needs to work at scale.
